# 基于 RAG 与 LLM 的企业数据智能分类分级系统 — 答辩讲解提纲

## 一、一句话说清楚

> 利用 RAG（检索增强生成）技术，将《个人信息保护法》《数据安全法》等法规文本向量化存入本地知识库，当企业对某个业务字段进行合规定级时，系统自动检索最相关的法律条文作为上下文，交由大模型进行思维链推理，最终给出 L1~L4 的分级结论，并提供全流程可溯源的法律依据。

---

## 二、解决的业务问题

企业数据合规的核心痛点：

| 痛点 | 现状 | 本方案 |
|------|------|--------|
| 定级靠人 | 法务/安全团队逐字段翻阅法规手册，效率低、标准不一 | 自动化语义检索 + LLM 推理，秒级出结果 |
| 法规更新 | 新法规发布后难以及时消化 | 上传 txt 即可增量入库，知识库即时更新 |
| 不可溯源 | 人工判定只给结论，说不出依据 | 每次推理附带检索到的 Top 3 法律条文原文 |
| 难以纠错 | 判错了就错了，缺乏纠错闭环 | 错题本机制：上传预设标准答案，命中后强制拦截 |

## 三、技术选型逻辑

### 为什么选 RAG 而不是微调？

1. 微调需要标注大量（字段, 级别）的样本 —— 本项目没有
2. 法规条文会更新 —— 微调模型知识固化，RAG 只需更新向量库
3. 合规审计要求可溯源 —— 微调是黑盒，RAG 天然携带检索依据
4. 四份法规总共约 8 万字，向量检索完全覆盖

### 为什么选 DeepSeek 而不是本地部署？

1. DeepSeek API 成本极低（约百万 token 几毛钱）
2. 本项目是可行性验证阶段，快速验证 > 基础设施投入
3. temperature=0.1 保证推理严谨性

## 五、核心 RAG 链路（一次完整推理的 6 个环节）

```
用户输入: 英文名=account_balance, 中文名=账户余额, 业务描述=客户账户资金余额

Step 1 ─ 错题本拦截
  │ 检查 "account_balance" 是否在 ERROR_LOG_CACHE 中
  │ → 命中则直接返回预设答案，跳过 LLM
  │
Step 2 ─ 向量检索
  │ 将 "账户余额 客户账户资金余额" 转为向量
  │ 在 Chroma 库中余弦相似度检索，取 Top 3
  │ → 返回 3 段最相关的法律条文原文
  │
Step 3 ─ 上下文组装
  │ 将 3 段法律依据 + 字段信息 填入 CoT Prompt 模板
  │
Step 4 ─ LLM 思维链推理
  │ DeepSeek 按 Step 1/2/3 逐步推理：
  │   Step 1 (语义分析): 账户余额反映个人财产状况
  │   Step 2 (合规比对): 对照《个保法》第28条，属敏感个人信息
  │   Step 3 (输出结论): 最终定级：L3
  │
Step 5 ─ 级别解析
  │ 正则匹配 "最终定级：L3" → 提取 L3
  │
Step 6 ─ 返回三件套
  │ (级别=L3, 推理过程原文, 检索到的法律依据)
  └────────────────────────────────────────
```

## 六、关键设计决策

### 6.1 Prompt 思维链设计

不是直接问「这个字段是什么级别？」，而是要求模型走三步：

- **Step 1（语义分析）:** 这个字段在业务中代表什么实体？
- **Step 2（合规比对）:** 对照检索到的法律条文，每一条怎么说？
- **Step 3（输出结论）:** 最终定级：LX

**好处:**
- 推理过程可审计（不是黑盒结论）
- 强制模型给出分析路径，减少幻觉
- 级别解析用正则匹配「最终定级：LX」，不依赖关键词猜

### 6.2 错题本机制（规则 + AI 混合架构）

**设计理念:** 纯 AI 有不确定性，纯规则覆盖不全；使用**规则优先 + AI 兜底**方案。

```
ERROR_LOG_CACHE = {
  "account_code": "L2",    # ← 人工确认过标准答案的字段
  "salary_info":  "L3",
}

逻辑：
  if 字段名 in 错题本:
      直接返回预设级别（0 成本，100% 准确）
  else:
      走完整的 RAG + LLM 推理
```

**实现的闭环:**
- 已纠正的错误不会再犯
- 新字段自动走 AI 推理
- 评测后发现的新错题，导出后下次上传即可生效

### 6.3 知识库增量更新

```
初始: 4 份法规 txt → RecursiveCharacterTextSplitter(chunk=500, overlap=50)
     → HuggingFaceEmbeddings → Chroma 持久化

运行时增量: 上传新法规 txt → 同样切分 → add_documents → persist
```

好处：知识库可以随时进化，无需重建整个向量库。

### 6.4 批量评测闭环

测试集 Excel 格式：

| 英文名 | 中文名 | 业务描述 | 标准答案 |
|--------|--------|----------|----------|
| salary | 工资   | ...      | L3       |

评测输出（多 Sheet Excel）：
- **Sheet 1 — 全量评测记录:** 含每条的大模型结论 + 推理理由 + 检索依据
- **Sheet 2 — 错题核查清单:** 大模型结论 ≠ 标准答案 的条目
- **Sheet 3 — 量化评估指标:** 准确率 + 每级的精确率/召回率/F1

**错题闭环:** 评测 → 导出错题本 → 人工复核 → 上传为错题集 → 下轮评测自动修正

## 九、当前不足与后续方向

1. **知识库覆盖面:** 目前仅 4 份法规，实际合规还需行业标准（金融、医疗等行业规范）
2. **Embedding 模型:** 当前用通用 sentence-transformer（MPNet），法律责任领域的语义匹配精度可能不如领域微调模型
3. **Prompt 泛化:** 少数情况模型不按「最终定级：LX」格式输出，正则回退逻辑能兜底但不够优雅
4. **性能:** 批量评测是串行调用，后续可考虑异步并发
5. **多轮对话:** 目前是单轮推理，未支持追问和澄清（如"这个字段还有别的上下文吗？"）
